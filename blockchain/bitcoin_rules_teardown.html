Where do Bitcoin's
rules actually live?

You asked two sharp things: are the rules in the software the servers download? and is there software besides Bitcoin Core? The answer to both is yes — and the why reveals how the whole system holds together.

We'll take it apart the same way we tear down any real build: what the rule is, what enforces it, what it costs, and the killer question — same rules, or different network?

A TEARDOWN OF THE PROTOCOL ITSELF
THE CORE IDEA

There is no rulebook. There is only code that runs.

YOUR QUESTION, ANSWERED

Yes — the rules live in the software each node downloads and runs.

There is no central server holding "the official rules of Bitcoin." The protocol is defined by what the running software accepts and rejects. Every full node has its own copy of the validation logic and checks every block and transaction against it independently.

  • The blockchain is the data — the chain of blocks, the ledger of who-paid-whom.
  • The consensus rules are the logic — the program that decides which blocks and transactions are valid.
  • A block that breaks a rule is rejected by every honest node, no matter who produced it or what they claim.

The rules aren't written on a tablet somewhere. They're whatever the software you're running will accept.

This is why the question "where are the rules" is deeper than it looks: the rules are replicated across thousands of machines, and agreement among them is what we call consensus.

WHAT A NODE CHECKS

The consensus rules, concretely.

When your node receives a block, it runs it past a checklist. If any item fails, the block is dropped. These are the load-bearing ones:

🧮
SUPPLY CAP
No more than 21 million BTC will ever exist. A block trying to mint extra coins is rejected.
💰
BLOCK REWARD SCHEDULE
The reward halves every 210,000 blocks (~4 years). A miner claiming too much is rejected.
⚒
PROOF-OF-WORK & DIFFICULTY
The block's hash must be below the current difficulty target, which retargets every 2,016 blocks to keep ~10-minute spacing.
✅
SIGNATURES & SCRIPTS
Every input must carry a valid signature unlocking the coins. No valid signature, no spend.
🚫
NO DOUBLE-SPEND
Coins spent in one transaction can't be spent again. Each input references an unspent output.

These live in Bitcoin Core's C++ source (mostly in the validation code), and in every other implementation that wants to stay on the same network.

If one person edited their own node's code to allow a 22nd-million coin, what happens?
The whole network inflates — everyone now has extra coins
Nothing useful: every other node rejects their block, so they fork themselves off alone
Bitcoin Core's central server overrides them
Locked in. Every other node rejects that block, so the rule-breaker is left alone on a chain nobody else follows. There is no central server to override anyone — rejection by peers is the enforcement.

YOUR SECOND QUESTION

The other software besides Bitcoin Core

THE LANDSCAPE

Bitcoin Core dominates — but it was never the only client.

Bitcoin Core runs on the large majority of nodes, but several other implementations exist. The crucial split isn't the programming language — it's whether they enforce the same consensus rules.

⚙️
BITCOIN CORE
C++. The reference implementation and dominant client. When people say "the rules," they usually mean what Core accepts.
🪝
BITCOIN KNOTS
A derivative of Core (maintained by Luke Dashjr) sharing most of its codebase, with extra configuration options. Same consensus rules.
🐧
btcd
An independent full node written in Go — not derived from Core's C++. A genuinely separate codebase that targets the same rules.
📚
Bcoin / Libbitcoin
Other alternatives: Bcoin (JavaScript) and Libbitcoin (a C++ library/toolkit). Different code, same network — when they agree on the rules.
⚠️
XT / CLASSIC / UNLIMITED
Historically contentious clients from the ~2015–2017 scaling debates that proposed different rules (mainly bigger blocks). Now largely defunct.

Same rules, different code. Core, Knots, and btcd can all interoperate on one network — they accept and reject identical blocks, just implemented in different languages.

Agreement is the whole point. A subtle bug that makes one client validate differently from the majority would fork that client's nodes off the network — which is exactly why most operators run the most battle-tested implementation.

✓
Killer question: same network

Different software that enforces identical rules is still the same Bitcoin. The diversity is healthy: if one client has a bug, others can catch the divergence.

THE OTHER OUTCOME

When the rules diverge: forks

THE LINE THAT MATTERS

Change the rules, and you change the network.

Running alternative software is fine only if it enforces the same consensus rules. The moment a client intentionally changes a rule and enough people adopt it, you don't have "alternative Bitcoin software" — you have a separate blockchain and a separate currency.

The famous splits were not just "different software" — they were deliberate hard forks with changed rules: Bitcoin Cash (2017) and later Bitcoin SV (2018), both mainly raising the block-size limit, each creating its own chain and coin.

A client ships that allows 32 MB blocks instead of the standard limit, and a group adopts it. Same Bitcoin?
Yes — it's just a faster version of the same network
No — old nodes reject the big blocks, the chain splits, a new coin is born
Bitcoin Core decides whether to allow it
Locked in. Old nodes reject the oversized blocks, the two groups stop agreeing, and the chain splits into a new coin. That's a hard fork — different rules mean a different network.
THE TEACHING POINT

"Which software" is a smaller question than "which rules."

Your two questions are really one question seen from two sides. The rules live in the software — and there are many pieces of software. What keeps it all "Bitcoin" is not a central authority blessing one program, but thousands of independent nodes agreeing, block by block, on the same validation logic. Swap the language, keep the rules: same network. Keep the language, change a rule: new network.

Consensus isn't a vote you win. It's the set of blocks everyone's software already agrees to accept.
